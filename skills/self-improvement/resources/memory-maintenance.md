# Memory Maintenance

Trae 的原生 Memory 系统有明确的容量限制和管理行为：

- **每条容量上限**：Project 和 Global 各最多 20 条
- **自动淘汰**：达到 20 条后新增时，系统自动删除最不常用/最久未用的条目
- **没有批量管理 UI**：只能在设置中心逐条删除
- **记忆存储形式**：本地加密数据库，非文件系统明文
- **记忆注入方式**：新对话启动时，IDE 从数据库读取全部 user_scope + project_scope 条目，封装为 XML 结构注入到 Agent 的系统提示词中

这意味着：如果不主动管理，关键记忆可能被自动淘汰，低价值记忆可能占据空间。

## Project vs Global 判定标准

| 场景 | 范围 | 示例 |
|:----|:----|:------|
| 仅当前仓库有效 | `project` | "这个项目用 pnpm，不用 npm" |
| 个人长期偏好 | `user` | "用中文回复我"、"调用我 David" |
| 仅当前仓库的技术约定 | `project` | "后端用 FastAPI，前端用 Vue 3" |
| 跨项目的通用习惯 | `user` | "提交信息用 conventional commit 格式" |

**简单规则**：如果这条记忆换一个项目还能用，用 `user`；只能当前项目用，用 `project`。

## 需要主动清理的情况

出现以下任一信号时，在执行下一个主要任务前执行一次清理：

- 当前 scope 的记忆数 ≥ 18 条（接近 20 上限）
- 发现一条已过时的记忆（比如项目已从 npm 迁移到 pnpm，但记忆还写着"用 npm"）
- 发现一条重复或冗余的记忆（与另一条高度重叠）
- 连续 3 次以上没有触发任何记忆匹配（说明记忆未被使用）

## 清理流程

使用 `manage_core_memory` 工具，`action` 设为 `DELETE`，传入要删除的 `memento_id`。

### 清理原则

1. **优先删除**：
   - 过时的技术约定（已被项目实际行为取代）
   - 一次性教训（不会再次发生的环境问题）
   - 冗余重复（多条目可合并为一条）

2. **保留**：
   - 用户主动要求记录的偏好
   - 安全/配置相关的约束
   - 近期在对话中被引用过的记忆

3. **不确定时**：保留。宁可占用空间，不要丢失可能还有用的信息。

## 何时执行

- **创建前检查**：在调用 `manage_core_memory ADD` 之前，如果当前 scope 记忆数 ≥ 18 条，先执行清理再新增。
- **定期维护**：作为 `finishing-a-development-branch` 的 Knowledge Promotion Gate 一起执行，在记录新教训前先清理过时记忆。

## 技术背景：记忆的存取方式

Agent 对记忆的实际感知受以下因素影响：

1. **会话启动时选择性注入**：新对话启动时，IDE 从本地加密数据库读取部分 user_scope + project_scope 条目，封装为 XML 结构注入到 Agent 的系统提示词中。**并非完整数据库转储**——注入策略可能是最近/N条/高频匹配的条目，具体选择机制由 IDE 端控制且不透明。
2. **静态快照**：同一对话内，后续通过 `manage_core_memory` 增删改的内容不会自动刷新注入的 XML 快照。
3. **会话间不共享状态**：不同对话（或不同 Agent 实例）可能收到不同的记忆子集。一个会话中创建的条目，另一个会话不一定能立即看到。
4. **重启不保证完整刷新**：Trae 重启后新会话注入的是重启时刻的快照，仍受选择策略影响，不保证是完整集。
5. **`manage_core_memory` 工具**：仅支持 ADD/UPDATE/DELETE 原子操作。没有 LIST/COUNT 操作，Agent 也无法通过工具验证自己看到的记忆集是否完整。

**关键结论**：Agent 从上下文中看到的记忆集**可能是不完整的子集**。Agent 无法自行验证完整性。"帮我整理记忆"时，Agent 不能声称"从上下文读取即为完整集"——必须先说明这个限制，并让用户通过设置页面确认实际总量。

## 整合审计

与"接近上限时清理"不同，整合审计是**主动的知识压缩**——不是因为空间不够，而是为了让记忆集保持高质量和低噪声。

建议在以下时机由用户主动提出，或 Agent 在对话中推测到后提醒用户执行一次：

- 对话中提到"记一下"、"记住"、"记一条"的频率明显增加（说明记忆碎片在膨胀）
- 项目发生过重大变更（技术栈更换、架构重构）
- 用户自己觉得"记忆好像有点乱"

### 用户触发流程（"帮我整理记忆"）

当用户说"帮我整理记忆"或类似意图时，按以下步骤执行：

1. **确认范围**：询问用户整理 Project 还是 Global（用户级）记忆？默认 Project。
2. **说明限制**：告知用户"我当前看到的记忆来自会话启动时注入的快照，可能不是完整的数据库。请帮我从设置 → 规则与技能 → Memories 确认一下实际条目总数。"
3. **等待用户确认总量**：用户告知实际条数后，与我上下文中的记忆集数量对比。如果两者一致，继续；如果不一致，以用户告知的总数为准，并让用户手动将未出现的条目复制给我。
4. **执行审计**：将获取到的完整记忆集逐条列出，执行下方的审计五步流程
5. **列出修改建议**：说明每条建议合并或删除的原因，让用户确认后再执行
6. **执行变更**：用 `manage_core_memory` 的 DELETE 和 ADD 执行确认后的操作
7. **报告结果**：说明清理了哪些、合并了哪些、最终保留几条。同时提醒用户：变更将在下次新对话中生效

### 审计五步流程

1. **枚举**：逐条列出当前 scope 的所有记忆，包括 memento_id、title、content 摘要
2. **分类**：逐条打标——保留 / 合并 / 删除
   - **合并**：多条记忆描述同一主题的不同方面（如"路由规则"和"护栏红线"），合并为一条综合条目
   - **删除**：已被后续变更取代、一次性教训、从未被匹配过的低价值记忆
3. **合并**：对标记为"合并"的条目，用一条新的综合条目替代，content 覆盖所有原有要点
4. **删除**：删除标记为"删除"和被合并替代的旧条目
5. **验证**：确认删除后的总数，确保关键决策信息未被丢失
