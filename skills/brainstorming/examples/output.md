# 示例输出

## 期望行为

- 先判断任务是否真的需要设计阶段，而不是机械触发长流程。
- 如果需求仍模糊，先逐步澄清目标、约束和成功标准，每次只问一个问题。
- 在形成设计建议前，给出 2-3 种方案、权衡和推荐。
- 输出结果默认是固定骨架的内联设计摘要；只有在确有价值时才写规格文件。
- 如果设计已经收敛且下一步需要详细计划，优先衔接 `writing-plans`；只有当前环境缺少该 skill 时才直接在当前对话中给计划。
- 如果环境支持且问题本身偏视觉，可先单独提议使用 visual companion；如果不支持，则继续纯文本流程。

## 固定摘要骨架

```markdown
设计讨论已收敛。

**目标：** [一句话]
**关键约束：**
- [若无则写 `- 无`]

**方案对比：**
- 方案 A：[一句话 + 取舍]
- 方案 B：[一句话 + 取舍]
- 方案 C：[如无则省略]

**推荐方案：**
- [推荐理由]

**关键结构：**
- [模块 / 页面 / 数据流 / 边界]

**风险或待确认点：**
- [若无则写 `- 无`]

**下一步：**
- [进入 writing-plans / 直接实现 / 写规格文件 / 结束在设计结论]
```

## 不期望行为

- 对清晰的小改动仍强行走 brainstorming。
- 直接跳到实现或计划，跳过澄清和方案比较。
- 在已经存在 `writing-plans` 的环境里仍忽略它，或者反过来硬编码依赖一个当前环境中不存在的计划类 Skill。
- 因为 visual companion 不可用而阻塞整个流程。
