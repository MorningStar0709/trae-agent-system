# 示例输出

## 期望行为

- 先确认当前环境确实支持实现型子代理，且任务之间基本独立，再进入子代理驱动执行。
- 控制者先提取完整任务文本和上下文，再分派实现者；不要把“自己去读计划文件”甩给子代理。
- 每个任务严格遵循 `实现 -> 规格审查 -> 代码质量审查` 的顺序，不跳过任何一环。
- 任一环节发现问题时，返回对应修复环节，不带问题进入下一个任务。
- 最终回复使用固定汇报骨架，明确任务进度、审查结果、验证结果和阻塞或风险。

## 固定回复骨架

```markdown
子代理驱动执行已完成。

**计划：** `docs/superpowers/plans/<filename>.md`
**执行结论：** `全部完成` | `部分完成` | `阻塞`
**任务进度：** `<completed>/<total>`

**任务结果：**
- 任务 N：`通过` | `阻塞` | `需继续修复`

**审查结果：**
- 规格合规性：[通过 / 问题摘要]
- 代码质量：[通过 / 问题摘要]

**验证结果：**
- [测试 / lint / 构建 / 手动检查]

**阻塞或风险：**
- [若无则写 `- 无`]

**下一步：**
- [继续修复 / 切换到 executing-plans / 等待用户决定 / 进入提交流程]
```

## 不期望行为

- 在无法确认实现型子代理可用时仍强行启用本 skill。
- 任务强耦合却仍按子代理独立任务拆分推进。
- 跳过规格审查或代码质量审查。
- 某个任务仍有未解决问题，却继续进入下一个任务。
